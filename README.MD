# Практическое задание 3.

Выполнила Петрова Кристина ЭФБО-06-23.

В данном задании было реализована конфигурация приложения из трех источников: из файла, переменных окружения, командной строки.
Приоритет источников: CLI > ENV > FILE
Также была реализована валидация: 
- проверка порта
- проверка режима работы 
- проверка формата trusted origins
- проверка лимитов
При ошибке происходит завершение работы.

Поддерживаются два режима:
- educational: подробные логи, упрощённое поведение
- production: минимальные логи, строгие ограничения

Использовалась базовая защита веб-сервиса через CORS, ограничение лимитов, защитные заголовки (X-Content-Type-Options: nosniff, X-Frame-Options: DENY, Cache-Control: no-store)

Также был реализован DI-контейнер с поддержкой жизненных циклов: синглтон (один на приложение), транзиент (новый при каждом создании) и скопд (один на запрос).

Для теста параллельности используется многопоточность и отправляется несколько одновременных запросов к эндпоинту /di-test, который получает экземляры сервиса, сравнивает их и увеличивает счетчик.

Выводы:
- Синглтон - увеличивает счетчик глобально, состояние "протекает между запросами", что может привести к проблемам с многопоточностью.
- Скопд - вкутри запроса объект один, что безопасно для веб-приложений. Он не вызывае таких проблем как у синглтона из-за своей изоляции.
- Транзиент - изоляция состояния, но может привести к увеличению использования ресурсов при создании крупных объектов.

Сравнение:
• Состояние: Синглтон хранит состояние глобально (возможно, нежелательно), Транзиент изолирован (но может быть ресурсоемким), а Скопд обеспечивает безопасность состояния на уровне запроса.
• Использование ресурсов: Синглтон экономит ресурсы, Транзиент использует больше ресурсов, а Скопд занимает промежуточное положение.
• Безопасность: Скопд является наиболее безопасным вариантом для веб-приложений, так как он предотвращает проблемы с состоянием между запросами.

Выводы:
1. Ранняя валидация конфигурации позволяет обнаружить ошибки до запуска и снижает риски отказа системы.
2. Использование нескольких источников конфигурации повышает гибкость приложения.
3. Ограничение CORS и rate limiting защищают сервис от злоупотреблений.
4. Параллельные запросы наглядно демонстрируют проблемы общего состояния.
